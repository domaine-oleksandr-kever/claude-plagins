# Validate Brand Config and CSS Tokens

Audit a brand's CSS files (`@colors.css`, `@theme.css`, `@typography.css`, `@buttons.css`) against its `design-token.manifest.json`, and cross-check `config.json` so brand settings (`fontMapping`, `fluidTypography`, `buttonVariants`) are reflected in the CSS. Report any values that are out of date or any config-driven expectations missing from the CSS.

**This skill is read-only by default.** It does not modify files unless the user explicitly approves updates in Step 5.

The detailed per-file audit rules, resolution algorithms, config cross-check tables, report format, and edge cases live in **`validate-brand-config-and-tokens/REFERENCE.md`**. This file is the high-level process.

## Step 1: Get the Brand Slug (MANDATORY)

Use the `brand_slug` argument if it was provided. **If it wasn't, ask the user first:**

> Which brand slug should I validate? (e.g., `estee-lauder`, `mac`)

Then verify these files exist before proceeding:

- `multi-brand/brands/{slug}/design-token.manifest.json`
- `multi-brand/brands/{slug}/config.json`
- `multi-brand/brands/{slug}/styles/@colors.css`
- `multi-brand/brands/{slug}/styles/@theme.css`
- `multi-brand/brands/{slug}/styles/@typography.css`
- `multi-brand/brands/{slug}/styles/@buttons.css`

If any file is missing, report which files are missing and stop.

## Step 2: Read All Source Files

Read these files in full:

1. `multi-brand/brands/{slug}/design-token.manifest.json` — source of truth for token _values_
2. `multi-brand/brands/{slug}/config.json` — brand config; must stay in sync with how CSS is structured (see **3e**)
3. `multi-brand/brands/{slug}/styles/@colors.css` — current color tokens
4. `multi-brand/brands/{slug}/styles/@theme.css` — current theme tokens (spacing, radius, fonts, button vars)
5. `multi-brand/brands/{slug}/styles/@typography.css` — current typography styles
6. `multi-brand/brands/{slug}/styles/@buttons.css` — current button component styles
7. `snippets/@color-schemes.liquid` — defines which `--color-*` and `--button-*` variables the storefront needs

The manifest is generated by `multi-brand/scripts/pull-figma-tokens.ts`. See the **Figma extended collections** note in REFERENCE.md for why `fontMapping` keys must match the resolved manifest font names, not leftover base names.

## Step 3: Run Validation Checks

For each CSS file, compare current values against expected values derived from the manifest using the **Resolution Algorithms** in REFERENCE.md. Track every discrepancy as a finding. The full per-check tables are in REFERENCE.md:

- **3a — `@colors.css`**: semantic, inverse, accent/tertiary, utility, grey scale, and missing variables vs `@color-schemes.liquid`.
- **3b — `@theme.css`**: font families, the full spacing scale (fluid LIMITED_DESKTOP), border radius, button color tokens.
- **3c — `@typography.css`**: font-size, line-height, letter-spacing, font-family (via `fontMapping`), font-weight, text-transform across base / components / static / `@utility`.
- **3d — `@buttons.css`**: button typography, uppercase, border-radius and padding fallbacks, all variants present.
- **3e — `config.json` vs brand CSS** (MANDATORY): `key`, `designTokenManifest`, `fontMapping`, `fluidTypography`, `buttonVariants`. Run this so `config.json` changes are not silently ignored by the CSS.
- **3f — New manifest tokens missing from CSS** (MANDATORY): cross-check every manifest key against the CSS so newly synced Figma tokens surface instead of being silently dropped. A "new token" is always `⚠️ Review`, never `❌`.

Config controls _structure_ and _font references_; the manifest wins for numeric/token values. See REFERENCE.md "Source-of-truth precedence".

## Step 4: Generate Report

Present findings using the structured report format in REFERENCE.md (Summary table → Detailed Findings per file → Missing Variables → New Manifest Tokens → Extra Variables). Key report rules:

- Only show detailed rows for mismatches, unless there are 0 issues (then show a few representative matches to confirm the audit ran).
- Include the **config.json vs CSS** subsection whenever **3e** finds issues; otherwise omit or note "no config/CSS drift".
- Include **New Manifest Tokens** whenever **3f** finds entries with no CSS surface; otherwise omit or note "no new manifest tokens".
- For fluid values, show the extracted mobile/desktop numbers, not the full `min(max(...))` expression.
- Judgment-based tokens (accents, background-tertiary, utility colors) are `⚠️ Review`, not `❌ Mismatch`.

## Step 5: Ask to Update

After the report, if there are any mismatches or new tokens (manifest/CSS, config/CSS from **3e**, and/or new manifest tokens from **3f**):

> **{n} issue(s) found** (tokens, config drift, and/or new manifest tokens). Would you like me to update the files?
>
> - **Yes, update all** — align CSS with the manifest **and** with `config.json` per **3e**, and declare any new tokens from **3f** in `@colors.css` / `@theme.css` (without wiring them into `@color-schemes.liquid` or components — that stays a manual decision)
> - **Let me choose** — I'll list each change and you can approve/reject individually
> - **No, just the report** — no changes will be made

**Yes, update all:** apply fixes to the CSS files (and `config.json` only when the drift is a config error — e.g. wrong `key` or manifest path — not to paper over intentional CSS). For **3f** new tokens, add the declaration with the resolved manifest value plus a short comment noting it is declared for surface parity and not yet consumed. Then verify the build (see below) and show a summary of changes.

**Let me choose:** present each mismatch as a numbered item with current → expected; ask which to apply (e.g. "1, 3, 5" or "all except 2"); apply selected fixes, then verify build.

**No:** end the validation. The report stands as documentation.

### Build verification

When updates are applied, run `npx vite build` with `BRAND` set, e.g. `BRAND={slug} npx vite build` (or define `BRAND` in a root `.env` / `.env.production`; the multi-brand Vite plugin loads these via `loadEnv`, with shell `BRAND` overriding file values) to verify the build passes.

## Important Notes

- **Read-only by default** — no file modifications unless the user approves updates in Step 5.
- **Judgment-based tokens** (accents, background-tertiary, utility colors) are flagged for review, not auto-fixed.
- **Structural changes** (missing `@utility` blocks, missing variants, wrong CSS structure) are reported but should be fixed via a full CSS regeneration, not patched inline.
- **The manifest is the source of truth** for resolved token _values_; **`config.json` is the source of truth** for how those values are _wired_ into CSS. See REFERENCE.md "Source-of-truth precedence".
