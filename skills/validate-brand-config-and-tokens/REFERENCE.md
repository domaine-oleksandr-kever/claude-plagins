# Brand config & CSS token validation — detailed reference

The per-file audit rules, resolution algorithms, config cross-check tables, report format, and edge cases for `validate-brand-config-and-tokens`. SKILL.md is the high-level process; this file is the source of truth for the checks.

## Source-of-truth precedence

- **The manifest is the source of truth** for resolved token _values_ (colors, spacing numbers, typography sizes from Figma, etc.).
- **`config.json` is the source of truth** for how those values are _wired_ into CSS (font slots via `fontMapping`, fluid vs static typography via `fluidTypography`, which button variants exist).
- If CSS disagrees with the manifest on a number, the manifest wins. If CSS disagrees with `config.json` on mapping or structure, the config wins — unless you document an intentional override as `⚠️ Review`.

## Figma extended collections (manifest generation)

The manifest is generated by `multi-brand/scripts/pull-figma-tokens.ts`, which must merge REST `variableOverrides` on extended collections (fonts, color schemes, responsive), not only each variable's `valuesByMode`. If that merge is missing, manifests mirror the base EDS (e.g. `Inter` for primary) while Figma's UI and DTCG exports (e.g. `Fonts.tokens.json`) show brand overrides such as `Roboto Mono`. After a correct pull, `fontMapping` keys must match the **resolved** names in `manifest.fonts` (`Font Families/Primary/Name`, etc.) — not leftover base font names.

## Resolution Algorithms

Use these to derive the "expected" value from the manifest.

### Color Resolution

The manifest's `colorSchemes` values are alias strings that reference `colorPrimitives` (or other scheme entries).

1. Look up the value string in `colorPrimitives` — if found, use that hex value.
2. If not in `colorPrimitives`, look up in the same color scheme mode — if found, recurse.
3. If the value already starts with `#` or `rgba`, use it directly.
4. Convert the final hex to `rgba(R, G, B, A)` format.

### Responsive Alias Resolution

Many values in the manifest's `responsive` section are string aliases. Follow the chain until you reach a number.

Example: `"Button/Radius": "Border Radius/SM"` → look up `"Border Radius/SM"` → `4`.

### Fluid Sizing Formula (LIMITED_DESKTOP)

Standard (mobile ≤ desktop):

```
min(max(calc({mobile} * 1px), calc(calc(100vw / 1440) * {desktop})), {desktop}px)
```

Decreasing (mobile > desktop):

```
clamp({desktop}px, calc({mobile}px - (100vw - 390px) * {factor}), {mobile}px)
```

where `factor = (mobile - desktop) / (1440 - 390)`.

### Typography Resolution

- **Font family:** `fonts["Heading 4XL/Font Family"]` → follow alias chain (e.g. `"Font Families/Primary/Name"` → `"Inter"`).
- **Font weight:** resolve style name to numeric weight — thin=100, extralight=200, light=300, regular=400, medium=500, semibold/semi bold=600, bold=700, extrabold=800, black=900.
- **Line-height:** compute `lineHeight / fontSize` as decimal (e.g. 56/48 = 1.17).
- **Letter-spacing:** compute `letterSpacing / fontSize` in em (e.g. 0.3/48 = 0.00625em). Omit if 0.

### Font family → CSS variable (`fontMapping`)

After resolving a style's font to a concrete name (e.g. `"Inter"`), look up that name in `config.json`'s `fontMapping`. The value is the slot label: `Primary`, `Secondary`, or `Tertiary` (case as in config). Generated CSS should use `font-family: var(--font-family-{lowercase-slot}, '{ConcreteName}')` — e.g. `"Inter": "Primary"` → `var(--font-family-primary, 'Inter')`.

If the manifest uses `"---"` for a font family name, that slot is unused: omit `--font-family-*` for that slot in `@theme.css` and do not expect mapping entries for it.

## 3a: Validate `@colors.css`

Check every `--color-*` variable inside the `@theme { }` block:

| Check | How |
| --- | --- |
| **Semantic colors** | Resolve each `colorSchemes.default` entry through the alias chain. Compare the resulting rgba against the CSS variable value. |
| **Inverse colors** | Same for `colorSchemes.inverse` entries mapped to `*-inverse` variables. |
| **Accent / tertiary** | Verify `--color-accent-01`, `--color-accent-02`, `--color-background-tertiary` and their inverse variants exist and are reasonable. |
| **Utility colors** | Verify `--color-green`, `--color-yellow`, `--color-blue`, `--color-orange`, `--color-light-blue`, `--color-white`, `--color-black` exist. Cross-check green/blue against `Positive/Positive` and `Info/Info` scheme entries. |
| **Grey scale** | Verify `--color-grey-100` through `--color-grey-600` exist and are ordered lightest → darkest. Cross-check values against grey primitives. |
| **Missing variables** | Cross-check against `@color-schemes.liquid` — every `--color-*` variable set there must have a default in `@colors.css`. |

## 3b: Validate `@theme.css`

### Font families

| Check | How |
| --- | --- |
| `--font-family-primary` | Compare against `fonts["Font Families/Primary/Name"]` in manifest. |
| `--font-family-secondary` | Compare against `fonts["Font Families/Secondary/Name"]`. Should be omitted if value is `"---"`. |
| `--font-family-tertiary` | Same for tertiary. |

### Spacing scale

For each `Space/*` key in the manifest:

1. Resolve the mobile and desktop values (follow aliases).
2. Compute the expected fluid value using the LIMITED_DESKTOP formula.
3. Extract the mobile and desktop numbers from the CSS fluid expression.
4. Compare: do the mobile and desktop numbers in the CSS match the manifest?

Check these spacing groups:

- `--spacing-none` through `--spacing-3xl` (from `Space/*`)
- `--spacing-sm-*` / `--spacing-lg-*` static pairs
- `--spacing-pagemargin`, `--spacing-pagegutter`, `--spacing-pagecolumns` (from `Page Grid/*`)
- `--spacing-icon-*` (from `Icon Size/*`)
- `--spacing-header-*` (from `Header/*`)
- `--spacing-button-*` sizing tokens (from `Button/LG/*`, `Button/MD/*`, `Button/SM/*`, `Button/Border/*`)
- `--spacing-forms-*` (from `Forms/*`)
- `--spacing-badge-*` (from `Badge/*`)
- `--spacing-carousel-*` (from `Carousel/*`)
- `--spacing-buttongroup-*` (from `Button Group/*`)

### Border radius

| Check | How |
| --- | --- |
| `--radius-none` through `--radius-rounded` | Compare against `Border Radius/*` tokens. Convert px to rem (÷16). |
| `--radius-forms-radius` | Resolve `Forms/Radius` alias chain → should match the corresponding `Border Radius/*` value. |
| `--radius-button-radius` | Resolve `Button/Radius` alias chain → same. |
| `--radius-badge-radius` | Resolve `Badge/Border Radius` alias chain. |
| `--radius-modal-radius` | Resolve `Modal/Radius`. |

### Button color tokens

For each button variant (primary, secondary, tertiary, and their inverse counterparts):

1. Resolve the manifest's `colorSchemes.default` button entries (e.g. `"Button Primary/Text"`, `"Button Primary/Background"`, etc.) through the alias chain.
2. The CSS should reference `var(--color-*)` variables rather than hardcoded rgba — verify the referenced variable resolves to the same color.
3. Check that hover, disabled, and focus states are present and map correctly.

## 3c: Validate `@typography.css`

For each typography style in the manifest (Heading 4XL → h1, Heading 3XL → h2, etc.):

| Property | How to check |
| --- | --- |
| **font-size** | Extract mobile/desktop numbers from the fluid `min(max(...))` expression. Compare against manifest's `Typography/{style}/Font Size` for both modes. |
| **line-height** | Extract the decimal line-height. Compare against `Typography/{style}/Line Height` ÷ `Font Size` for the mobile base and the `@media (min-width: 1024px)` override for desktop. |
| **letter-spacing** | If present, verify it matches `Typography/{style}/Letter Spacing` ÷ `Font Size` in em. If 0 in manifest, should be absent or `0`. |
| **font-family** | Resolve `fonts["{style}/Font Family"]` through the alias chain to a concrete name, then apply `fontMapping` to get the expected `var(--font-family-*, '…')` (see **3e**). Verify CSS matches that expectation everywhere the style appears. |
| **font-weight** | Resolve `fonts["{style}/Style"]` to a numeric weight. Compare against CSS. |
| **text-transform** | Check `fonts["{style}/Uppercase"]` — if `true`, CSS should have `text-transform: uppercase`. |

Check these style groups:

- `@layer base` HTML tag styles (`h1`–`h6`, `p`, `blockquote`)
- `@layer components` fluid classes (`.h1`/`.heading-1` through `.h6`/`.heading-6`, `.body`/`.p`, `.caption`, `.pullquote`, `.utility`)
- `@layer components` static classes (`.text-heading-1` through `.text-heading-6`, `.text-body`, `.text-caption`, `.text-pullquote`, `.text-utility`)
- `@utility` directives (`h1`–`h6`, `p`, `text-body`)

## 3d: Validate `@buttons.css`

| Check | How |
| --- | --- |
| **Button typography** | Resolve `Typography/Button/*` from manifest. Compare font-size, font-family, font-weight, line-height, letter-spacing, text-transform against the `.btn` base class. |
| **Uppercase** | Check `fonts["Button/Uppercase"]` — if `false`, `.btn` should NOT have `text-transform: uppercase` or `letter-spacing` tracking. |
| **Border radius fallback** | The fallback value in `var(--button-border-radius, ...)` should match the resolved `Button/Radius` value. |
| **Padding fallbacks** | Verify `var(--button-padding-block, ...)` and `var(--button-padding-inline, ...)` fallback values match resolved `Button/LG/Padding Vertical` and `Button/LG/Padding Horizontal`. |
| **All variants present** | Check that every variant in `config.json`'s `buttonVariants` array has a corresponding `&[variant='...']` selector (see **3e** for full rules, including inverse and orphan selectors). |

## 3e: Validate `config.json` vs brand CSS

Treat `config.json` as controlling _structure_ and _font references_; the manifest still wins for numeric/token values. Flag any setting in config that the CSS does not implement (or any CSS pattern that contradicts config).

### `key` and `designTokenManifest`

| Check | How |
| --- | --- |
| **`key`** | Must equal the brand `{slug}` (e.g. `estee-lauder`). If not, report ❌ — wrong file or copy-paste error. |
| **`designTokenManifest`** | Must point at this brand's manifest, typically `multi-brand/brands/{slug}/design-token.manifest.json`. If it points elsewhere, report ❌. |

### `fontMapping`

| Check | How |
| --- | --- |
| **Theme font slots** | For each `"FontName": "Slot"` in `fontMapping` (`Slot` = `Primary` \| `Secondary` \| `Tertiary`), `@theme.css` must define `--font-family-{lowercase-slot}` with the quoted family `'FontName'` matching the key (same spelling the manifest resolves to). |
| **Manifest slots covered** | For each active manifest slot (`Font Families/Primary/Name`, etc.) whose resolved name is not `"---"`, that resolved name must appear as a **key** in `fontMapping`, and the mapped slot must match the manifest slot (Primary name → `Primary`, etc.). If the manifest introduces a new concrete font used in typography but missing from `fontMapping`, report ❌. |
| **Typography + buttons use mapping** | For each mapped typography style (see **3c** table) and for **Button** typography: resolve font family to a concrete name; look up `fontMapping[name]` → expected `var(--font-family-{slot}, 'name')` in `@typography.css` (base, components, `@utility`) and on `.btn` in `@buttons.css`. Mismatched variable, wrong slot, or raw `font-family` without the expected `var()` → ❌. |
| **Orphan mapping keys** | A `fontMapping` key that never appears as a resolved font from the manifest (typography + `Font Families/*/Name`) → ⚠️ Review (stale config). |

### `fluidTypography`

Allowed values: `false` or the string `"LIMITED_DESKTOP"`.

| Check | How |
| --- | --- |
| **Typography font-size strategy** | If `"LIMITED_DESKTOP"`: `@typography.css` and `.btn` **font-size** should use the fluid `min(max(calc(...` pattern for sizes (aligned with manifest mobile/desktop), not static-only `rem` typography. If `false`: typography **font-size** should use static `rem` and, when mobile ≠ desktop, `@media (min-width: 1024px)` overrides — not the typography fluid `min(max(...` formula. Spacing in `@theme.css` stays fluid regardless; do not use spacing to infer this flag. |
| **`@utility` blocks** | `@utility` font-size rules must follow the same fluid vs static rule as `@layer base` / matching component classes. |
| **Mismatch** | Config says fluid but CSS typography is static-only (or the reverse) → ❌; partial / inconsistent files → list each location. |

### `buttonVariants`

| Check | How |
| --- | --- |
| **Config → CSS** | For each entry `v` in the `buttonVariants` array, `@buttons.css` must include `&[variant='{v}']`. If the brand ships inverse styles (as in generated themes), also expect `&[variant='{v}-inverse']` for each `v`. Missing selector → ❌. |
| **CSS → config** | Collect variant strings from `&[variant='...']` in `@buttons.css`. Any variant (excluding duplicates / shared wrappers) that is **not** listed in `buttonVariants` and is not explainable as a standard inverse of a listed variant → ⚠️ Review (stale CSS or config missing an entry). |

## 3f: Detect new manifest tokens missing from CSS

Manifest syncs (`pull-figma-tokens`) can introduce **new** keys that don't yet have a corresponding CSS variable. These are not value mismatches — they are missing token surfaces, so 3a–3d won't flag them. Cross-check every manifest key against the CSS so new entries from Figma surface in the report instead of being silently dropped.

| Check | How |
| --- | --- |
| **New scheme entries** | For every key in `colorSchemes.default` (and the matching key in `colorSchemes.inverse`), derive the expected CSS variable name in kebab-case (e.g. `Accordion/Border` → `--color-accordion-border`; `Accordion/Border: Active` → `--color-accordion-border-active`; inverse counterpart adds the `-inverse` suffix). If the variable is not declared in `@colors.css`, report ⚠️ New token. |
| **New responsive keys** | For every key in `responsive.desktop` / `responsive.mobile` that maps to a token group already declared in `@theme.css` (`Space/*`, `Border Radius/*`, `Icon Size/*`, `Components/*`, `Sections/*`), check whether the corresponding `--spacing-*` / `--radius-*` / `--components-*` variable exists. If not, report ⚠️ New token and include the resolved fluid expression (using the LIMITED_DESKTOP formula) so the report shows what would be added. |
| **New font slots** | If `Font Families/{Slot}/Name` resolves to a concrete name (not `"---"`) and no `--font-family-{slot}` is declared in `@theme.css`, report ⚠️ New token plus the matching `fontMapping` gap (see **3e**). |
| **Default value** | For each new token, resolve the manifest value through the alias chain (colors → rgba; responsive → fluid expression) so the report can show the value that should be added. |

A "new token" is always ⚠️ Review rather than ❌ — declaring the variable in `@theme.css` / `@colors.css` is mechanical, but wiring it through `@color-schemes.liquid` or a component snippet is a design decision that this audit does not auto-resolve. Propose the addition; do not assume the consumer already exists.

### Report rules

- Only show detailed rows for items with mismatches, unless there are 0 issues (then show a few representative matches to confirm the audit ran).
- Include **config.json vs CSS** detailed rows whenever **3e** finds issues; if **3e** is clean, omit that subsection or note "no config/CSS drift".
- Include **New Manifest Tokens** whenever **3f** finds entries with no CSS surface; if **3f** is clean, omit that subsection or note "no new manifest tokens".
- Group related issues together (e.g. all h1 issues in one block).
- For fluid values, show the extracted mobile/desktop numbers rather than the full `min(max(...))` expression.
- For `var()` references in button tokens, compare the referenced variable name and its resolved color — both must match.
- If a value requires judgment (accent colors, background-tertiary, utility colors), flag it as `⚠️ Review` rather than `❌ Mismatch`.

## Edge cases & notes

- **Read-only by default.** Do not modify files unless the user explicitly approves updates in Step 5.
- **Judgment-based tokens** (accents, background-tertiary, utility colors) are flagged for review, not auto-fixed. These require human verification against design intent.
- **Structural changes** (missing `@utility` blocks, missing variants, wrong CSS structure) are reported but should be fixed via a full multi-brand CSS regeneration rather than patched inline.
- **Build verification on update:** run `npx vite build` with `BRAND` set — e.g. `BRAND={slug} npx vite build`, or define `BRAND` in a root `.env` / `.env.production` (the multi-brand Vite plugin loads these via `loadEnv`, with shell `BRAND` overriding file values).
